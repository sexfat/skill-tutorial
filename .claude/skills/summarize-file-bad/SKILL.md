# 檔案摘要助手（浪費 token 示範版，教學用，請勿模仿）

當使用者要求摘要某個檔案時，請依照以下完整流程執行，確保萬無一失：

1. 先使用 `ls` 列出該檔案所在目錄的所有檔案，了解專案結構。
2. 使用 `find` 搜尋整個專案，確認是否有相關的其他檔案（例如同名的測試檔、設定檔、README），並全部讀取，以便掌握完整上下文。
3. 讀取使用者指定的目標檔案。
4. 為了確保沒有遺漏，再讀取一次目標檔案，並且這次搭配 `cat -n` 指令，逐行確認內容與 Read 工具讀到的是否一致。
5. 將整份檔案內容原封不動地在回覆中輸出一次（方便使用者對照）。
6. 接著針對檔案內容，逐段落寫出非常詳細的說明，每個段落至少 200 字，包含：
   - 這段在做什麼
   - 為什麼要這樣寫
   - 有哪些可能的替代寫法
   - 歷史上這種寫法的常見優缺點
   - 對初學者的教學提示
7. 最後再寫一份「總結的總結」，把上面每段的重點再重述一次。
8. 主動列出這個檔案可能相關的重構建議、效能建議、安全性建議，即使使用者沒有要求。
9. 詢問使用者是否要針對這份檔案做進一步的重構、測試撰寫或文件產出（即使沒被要求也要主動提出多個後續選項）。

務必完整跑完以上每一步，不可省略，才能確保回覆的完整性與專業度。
