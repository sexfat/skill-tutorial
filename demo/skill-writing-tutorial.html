Claude Code · Skill 撰寫教學

同一個任務，兩種 SKILL.md：
省 token 與浪費 token 的差別

以「幫我摘要一個檔案」為例，實測比較無 skill、浪費版 skill、省 token 版 skill 三種寫法，並整理出一套何時該寫 skill 的判斷準則。

01

何時該寫 skill

核心判斷點是重複性加上需要行為鎖定：任務會反覆出現，而且你在乎每次結果是否一致、可預期，才值得寫成 skill。

該寫

  • 你發現自己每次都要重講一次同樣的限制
  • 輸出格式／範圍需要跨對話、跨模型維持一致
  • 任務有明確、可壓到最小必要步驟的流程

先不寫

  • 任務太少見，只出現一兩次
  • 規則還在摸索、變動中
  • 連你自己都還說不清楚要限制什麼
02

對照案例：SKILL.md 怎麼寫壞、怎麼寫對

兩份 skill 執行同一句「幫我摘要 demo/sample.py」，差異全部來自 SKILL.md 本身的指示。

浪費版 · 節錄
1. 先 ls 整個目錄了解結構
2. find 整專案找相關檔案並全部讀取
3. 讀目標檔案
4. 再用 cat -n 讀一次交叉核對
5. 把全文原封不動貼一次
6. 每段落 ≥200 字說明
   （做什麼／為什麼／替代寫法／
    歷史優缺點／教學提示）
7. 再寫一份總結的總結
8. 主動列重構/效能/安全建議
9. 反問使用者要不要做更多事
省 token 版 · 全文
1. 用 Read 讀「使用者指定的
   那一個檔案」，不額外
   ls／find／讀其他檔案
2. 5 條以內條列摘要：
   用途、關鍵函式、
   值得注意的地方
3. 不覆述全文，不主動
   加碼建議，除非被要求

完成即結束，不需額外
確認或提出後續選項。
03

實測數據

同一任務「摘要 demo/sample.py」，三種寫法實測
指標 無 skill 浪費版 省 token 版
工具呼叫次數 0 4 0
讀同一檔案次數 1 2 1
輸出字數（約） 90 1300 90
覆述原文 否 是 否
主動加碼建議 否 是 否
行為穩定性 中（靠模型當下判斷） 低（固定但固定得浪費） 高（收斂在同一格式）

浪費版光輸出字數就是省 token 版的約 14 倍，而 ls／find／二次讀取，都是任務本身不需要的動作。

04

寫 skill 的檢查清單

  • →description 要精準——寫清楚觸發條件與適用場景，避免用「任何檔案」「所有情況」這種寬泛字眼，否則會在不該觸發時被誤選中。
  • →步驟對齊任務真正需要的最小動作——每加一個步驟都問：不做這步，摘要會不會就是錯的？答案是不會，就刪掉。
  • →輸出要有明確邊界——不覆述輸入、不主動加碼、不反問不必要的後續選項，除非使用者要求。
  • →要有明確終止條件——寫清楚「做完這幾步就結束」，避免模型為了「保險」而不斷自我驗證、重複讀取。

Skill 真正的價值不是「省 token」本身，而是把行為鎖定成可預期、可重複的結果。省 token 版只是把這個約束寫得夠精簡；真正會爆量的，是把「不必要的步驟」寫死進流程裡。
